# Helpcenter Doc Writing

使用本技能产出结构清晰、可执行、面向客户的帮助中心文档，帮助客户更好地理解和使用管理后台。

## When To Use

当用户提出以下任务时启用本技能：

1. 新写一篇帮助文档、FAQ、操作指南或场景说明。
2. 改写已有帮助文档，使其更清晰、更统一或更符合规范。
3. 根据产品信息、需求说明、发布内容整理为 helpcenter 文档。
4. 统一标题、摘要、步骤、注意事项、术语、语气和信息层级。
5. 将零散规则应用到文档审校、补全、重组和模板化输出中。

## Audience And Scope

1. 默认面向客户，而非内部团队。
2. 默认服务于“帮助客户更好使用管理后台”这一目标。
3. 输出中优先保留客户真正需要完成任务的信息，不写内部协作过程、推理或编辑备注。

## Entry Modes

本技能优先先判断用户属于哪一种入口，再决定后续处理方式：

1. `快速撰写`：根据需求、功能说明、PRD、发布内容、产品素材起草帮助文档。
2. `快速翻修`：对既有帮助文档进行重构、补全、统一风格和提高清晰度。
3. `增量更新`：基于已有帮助文档，按照功能迭代、页面变更、字段调整、端差异或权限变化，更新原文中的相关内容。

## Screenshot Handling

1. 若素材中已包含截图或图示，按图文结合规范组织内容。
2. 若素材中不含截图，也应先完成可上线的文字版结构；不要因为缺图停止写作。
3. 无截图时，可在必要位置预留“建议截图位置”或“建议图示类型”，但不要编造截图细节。

## Core Workflow

1. 先判断当前任务入口属于哪一种：`快速撰写`、`快速翻修`、`增量更新`。
2. 若用户未明确说明入口，优先先判断：
   - 是写一篇全新的文档。
   - 是翻修一篇旧文档。
   - 是在原有文档基础上增加或更新内容。
3. 再识别文档目标：用户要解决什么问题，目标读者是谁，文档类型是什么。
4. 判断当前任务更适合写成哪类文档：`新手指引`、`场景文档`、`角色文档`、`功能文档`、`FAQ`。
5. 若用户已提供原始材料，优先基于原文整理；不要补造产品能力、页面、字段或流程。
6. 若属于 `增量更新`，优先识别：
   - 哪些内容是新增的。
   - 哪些内容是替换旧内容的。
   - 哪些入口、字段、按钮、步骤、效果、端差异或权限发生了变化。
7. 先搭建信息结构，再写具体内容，确保标题和段落层级服务于任务完成。
8. 涉及详细写作规范时，读取 [writing-rules.md](references/writing-rules.md)。
9. 若规则之间冲突，优先保证事实准确、步骤完整、读者易懂。
10. 输出前检查是否存在术语不统一、步骤跳跃、前提缺失、标题空泛、页面不一致或内部视角表述。

## Document Type Decision

### 1. 新手指引

适用于首次接触产品、需要尽快跑通一个任务的用户。

优先目标：
- 降低理解门槛。
- 降低第一次操作门槛。
- 先帮助用户跑通最短路径，再补充更多细节。

### 2. 场景文档

适用于某一业务场景下，需要让客户快速理解“在这个场景中平台能提供哪些能力支持”的内容。

优先目标：
- 以客户业务流程为主线做导航。
- 串联不同能力与子文档。
- 不把主文档写成冗长的功能大全。

### 3. 角色文档

适用于说明某个角色在系统或直播中的职责、能力边界、端差异和权限限制。

优先目标：
- 先讲清角色是谁。
- 讲清角色默认能做什么、不能做什么。
- 讲清不同端的差异与额外授权能力。

### 4. 功能文档

适用于讲清单一功能是什么、为什么用、怎么用、会产生什么结果。

优先目标：
- 围绕单一能力展开。
- 先帮助用户建立功能认知，再进入操作步骤。
- 让用户知道配置后会看到什么结果。

### 5. FAQ

适用于用户已经遇到具体问题，希望尽快找到原因和解决方案的内容。

优先目标：
- 用问题导向组织内容。
- 先回答问题，再解释原因和处理方式。
- 若需要更长操作说明，分流到对应功能文档。

## Writing Priorities

默认按以下优先级处理：

1. 准确：不补造事实，不写未经提供的界面、路径、按钮、字段或结果。
2. 易懂：让首次接触该功能或场景的客户也能顺着理解。
3. 可执行：步骤具备前提、动作和必要结果，避免空泛指令。
4. 易扫读：合理拆分标题、列表、提示与说明，避免整段堆叠。
5. 一致性：术语、语气、结构、命名风格保持统一。

## Tone Preferences

默认使用克制、中性的帮助中心文风。

1. 优先使用“该功能用于……”“支持……”“可通过以下路径进入……”“开启后……”等客观表述。
2. 非必要时，少用“你”“我”“我们”等代入式表达，避免口吻过强影响文档感。
3. 除非用户明确要求更口语化、更引导式的表达，否则默认保持说明书式、帮助中心式语气。
4. 当使用第二人称能明显降低理解门槛时，可少量使用，但应控制频率，避免整篇文档反复出现“你可以……”“如果你需要……”。

## Translation Rules For Raw Materials

将原始产品素材转成帮助中心文档时，默认做以下转换：

1. 把“功能存在”改写成“客户在什么情况下会用、如何使用”。
2. 把内部模块树改写成客户任务路径或业务路径。
3. 把字段堆叠改写成可执行步骤或字段说明。
4. 把宣传式表达收敛为帮助中心式表达，先讲功能，再讲价值。
5. 把编辑备注、缺失说明、内部注释留在草稿层，不直接进入对外文档。

## Field Definitions

当文档包含“功能说明 / 使用目的 / 使用场景”等字段时，默认按以下边界撰写：

1. `功能说明`：说明这个功能是什么、能做什么，通常用 1 到 2 句话写清楚；重点写能力本身，不写“为什么好”。
2. `使用目的`：说明用户在什么情况下会需要使用该功能；重点写“什么时候会用”，不重复功能说明。
3. `使用场景`：说明该功能常见于哪些业务场景或业务情境；使用场景应比具体操作更高一层，不要写得过细，不要把功能用途换一种说法重复一遍。
4. 写“使用场景”时，优先使用业务场景描述，例如“私域运营”“教育培训”“活动运营”等；避免写成“适用于支持某某功能的场景”这类同义改写。

## Markdown Output

默认输出应可直接粘贴到 Markdown 编辑器中使用。

1. 优先输出标准 Markdown 结构，包括标题、列表、表格、引用、加粗和图片语法。
2. 若原始素材已包含图片链接，应原样保留图片链接，并继续使用 Markdown 图片语法。
3. 不输出依赖特定编辑器的富文本格式，不混入无法直接粘贴的说明性包装。
4. 若用户未要求落盘，默认直接在会话中返回可粘贴的 Markdown 最终版。

## Delivery Preference

当用户已明确要求同步到 GitHub 时，默认在输出完成后直接提交并推送当前相关改动，无需再次单独确认。

1. 优先只提交与当前文档或规则更新直接相关的改动。
2. 推送前仍需避免误带无关改动；若工作区存在明显冲突或无法安全区分的修改，再单独提示用户。
3. 若用户另行说明“只返回内容、不落盘”或“不要推送”，则以用户最新要求为准。

## Output Expectations

根据任务内容，优先输出以下一种或多种：

- 完整帮助文档正文。
- 优化后的重写版本。
- 建议标题与摘要。
- 推荐目录结构。
- 缺失信息清单（仅在确实无法安全补全时列出）。

## Guardrails

1. 不把面向内部团队的推理、猜测、待办、缺失备注或流程直接写进面向客户的文档。
2. 不使用“显然”“只需”“简单几步”等弱化复杂度的表达，除非用户明确要求轻口吻。
3. 不为了语言顺滑而牺牲操作准确性。
4. 如果原始信息不足以支撑具体步骤，明确标注缺失点，不编造。
5. 当用户明确提供风格、模板或固定文案时，以用户要求优先。
6. 文档中的功能名称、顺序、字段名应尽量与实际页面保持一致，避免用户无法通过文档定位界面。
7. 当主文档属于“场景文档”或“导航型文档”时，不要把每个子功能都展开成完整教程；优先挂到对应帮助文档。

## Reference
