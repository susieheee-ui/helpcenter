# Writing Rules

这里用于沉淀帮助文档撰写规范。优先保证文档面向客户、可执行、易扫读。

## Global Rules

- 帮助中心文档默认面向客户，目标是帮助客户更好地使用管理后台。
- 优先使用客户视角，不用内部协作视角。
- 不补造页面、字段、按钮、结果、权限或业务规则。
- 若素材不足以支撑具体步骤，应明确缺失点，而不是编造。
- 文档中的功能名称、字段名、入口顺序尽量与实际页面一致。
- 先组织结构，再写细节；不要一上来堆功能点或堆截图。
- 优先写“客户要完成什么”，再写“平台有什么”。
- 有截图时按图文结合规范写；无截图时先完成文字版结构，并在需要时预留建议截图位。

## Audience And Intent

写作前先判断：

- 目标读者是谁：首次使用者、已有经验用户、遇到问题的用户、某个角色用户。
- 用户当前要完成什么：首次跑通任务、理解单一功能、理解业务场景、确认角色边界、解决具体问题。
- 这篇文档最重要的产出是什么：跑通流程、建立认知、找到对应能力、理解权限、排查问题。

若这些判断不清，先通过文档类型决定结构。

## Structure

通用结构原则：

- 标题要直接说明用户能完成什么，不要空泛。
- 首段优先说明这篇文档帮助用户完成什么，而不是写内部目的说明。
- 正文结构优先服务于任务完成，不服务于后台模块罗列。
- 一个章节只承载一类信息：功能认知、步骤操作、效果展示、FAQ、角色限制等不要随意混写。
- 长文档优先分层：总览负责导航，子文档负责展开细节。

## Style And Tone

- 用词直接、克制、面向帮助中心，不用宣传口吻。
- 先讲功能做什么，再讲业务价值；避免堆叠“提升、增强、赋能”类空泛描述。
- 操作说明尽量以动词开头，如“进入”“选择”“上传”“点击”“开启”。
- 不使用“显然”“只需”“简单几步”等容易低估用户困难的表达。
- 当步骤包含较多配置项时，可主动降低门槛，例如：
  - 如果你暂时不确定如何选择，可先完成基础配置，后续再根据实际需求调整。
  - 如果你只是想先快速跑通流程，先完成必填项即可。
- 关键按钮、功能名、标签建议使用 `【】` 包裹，帮助用户快速定位。

## Screenshot And Annotation Rules

- 以“文字在上、截图在下”为默认顺序，避免先丢图再让用户自己找。
- 核心按钮、关键流程、关键状态需要标注。
- 截图中不得出现客户数据、个人账号、敏感信息；必要时先脱敏。
- 截图避免带入无关桌面和杂乱背景。
- 标注线条、箭头、文字大小应统一，优先细边框而非粗边框。
- 单一功能点说明适合用聚光灯式定位。
- 长路径操作、跨页面跳转、新弹窗链路更适合使用 GIF。
- 图示仅在确有帮助时出现；不要堆图。
- 每张图都应有明确说明目的，避免出现“孤立截图”。

## Procedure Writing

- 步骤尽量按实际操作顺序排列。
- 每一步尽量包含：起点、动作、必要对象、必要结果。
- 若步骤很多，优先拆成阶段，再写步骤。
- 不要把多个动作、多个字段、多个分支硬塞进一个长句。
- 当配置项很多时，优先区分“基础配置”和“按需补充配置”。
- 当某个功能是可选项时，应明确标记“可选”。
- 配置动作和结果展示要分开写：
  - “如何开启某个设置”属于操作步骤。
  - “开启后用户会看到什么”属于效果或结果说明。

## Accuracy And Boundaries

- 不把内部备注、待补充说明、缺失项注释直接写进对外文档。
- 不因为需要完整而补造不存在的页面流程或字段说明。
- 当原始材料只足够支持结构化整理，不足以支持详细教程时，应写成导航型文档，而不是伪装成详细教程。
- 若存在多端差异、权限差异、前置条件差异，应明确写出。
- 若存在对应帮助文档，主文档优先挂链接，不重复展开子文档全部内容。
- 暂无对应帮助文档时，可保留能力说明，但不要编造链接。

## Function Documents

适用场景：讲清单一功能是什么、为什么用、怎么用、配置后会产生什么结果。

### Recommended Structure

1. 功能概览
2. 操作步骤
3. 效果展示或结果说明
4. 后续动作（如结束、导出、查看结果）
5. 常见问题（可选）

### Function Overview

功能概览优先回答：

- 这个功能是什么。
- 这个功能能帮助客户解决什么问题。
- 客户在什么情况下会需要它。
- 客户从哪里进入这个功能。

功能介绍与使用目的、使用场景要区分：

- `功能介绍`：这个功能本身是什么，核心能力是什么。
- `使用目的`：客户通过这个功能想达成什么业务效果。
- `使用场景`：客户在什么业务情境下会用到它。

不要把结果当成功能介绍，也不要用行业标签替代使用场景。

### Function Writing Rules

- 先给功能概览，再进入操作步骤。
- 操作步骤应覆盖完整业务链路，而不只是“创建”动作。
- 当后台配置与前台效果都重要时，分开写。
- 当某个配置项会影响展示方式时，先在步骤中写“如何配置”，再在效果章节中写“配置后的展示结果”。
- 如果功能存在多种配置字段，可用表格或列表说明字段作用，但不要只列字段名。
- 当存在关键结果验证点时，应告诉用户配置成功后会看到什么。

### Common Review Checks For Function Docs

- 是否回答了“是什么 / 为什么用 / 什么时候用 / 从哪里进”。
- 是否把多个动作、多个字段压进一个句子。
- 是否把配置说明和展示结果混写。
- 是否存在图片堆叠但无文字说明。
- 是否遗漏适用对象、终端、版本等关键信息。
- 是否存在异常标题、残留排版、错误编号。

## Scenario Documents

适用场景：说明某一业务场景下，平台能如何支持客户完成整体业务目标。

### Core Principle

场景文档以“流程导航”为主，而不是以“功能罗列”为主。

目标是让客户快速理解：

- 这个场景是什么。
- 在这个场景下通常会经历哪些关键阶段。
- 每个阶段平台可以提供哪些能力支持。
- 需要更详细操作时，应该看哪篇对应帮助文档。

### Recommended Structure

1. 场景简介
2. 业务流程总览
3. 按业务阶段拆分的能力导航

### Scenario Writing Rules

- 优先按客户业务流程组织章节，不按后台菜单树组织章节。
- 主文档负责导航，不负责把所有子功能写成完整教程。
- 每个具体功能节点最好包含：
  - 这个功能能解决什么问题。
  - 在当前阶段为什么会用到它。
  - 对应帮助文档链接是什么。
- 若暂无对应帮助文档，可保留能力说明，但应避免假装已有详细文档。
- 不必增加“典型使用方式”或额外推荐阅读路径，若正文流程已足够清晰且已挂链接。
- 当一个能力跨多个阶段时，优先放在客户第一次真正会使用它的阶段，并在描述中补充后续用途。

### Common Review Checks For Scenario Docs

- 是否按客户流程写，而不是按产品模块写。
- 是否把主文档写成了功能大全。
- 是否给出了每个阶段对应的能力支持。
- 是否尽量为每个能力节点挂上对应帮助文档。
- 功能归类是否优先按“使用时机”，而非按产品类别。

## Starter Guides

适用场景：帮助第一次接触产品的用户尽快跑通一个任务，降低理解和操作门槛。

### Core Principle

新手指引的核心不是讲全，而是让新手觉得：

- 我看得懂。
- 我知道从哪里开始。
- 我可以先完成第一遍。

### Recommended Structure

1. 适用对象
2. 适用终端
3. 前置准备
4. 起始入口
5. 主要操作步骤
6. 完成后的下一步
7. 常见问题

若素材较少，也至少应覆盖：
- 你将完成什么
- 开始前准备什么
- 按最短路径完成操作
- 完成后的下一步

### Starter Guide Writing Rules

- 优先帮助用户建立信心，再补充细节。
- 优先给最短路径，不优先讲全量配置。
- 当一个步骤包含很多配置项时，应明确：哪些是先完成即可，哪些可后续调整。
- 尽量减少新手当下必须做的判断。
- 当用户暂时不理解某个配置项时，可以明确告诉用户先完成基础配置。
- 要让用户知道每一步在做什么、做完后会得到什么。
- 完成后的下一步应明确挂出对应文档，避免用户完成第一步后断掉。
- FAQ 应优先围绕首次使用时最容易卡住的问题，而不是泛泛补充阅读。

### Common Review Checks For Starter Guides

- 是否让首次使用者一眼知道能完成什么。
- 是否写清前置准备。
- 是否给出了最短路径。
- 是否把太多高级配置提前压给新手。
- 是否写清创建或配置成功后的结果和下一步。

## Role Documents

适用场景：说明某个角色在系统、直播或产品中的职责、登录端、默认能力、授权能力和限制。

### Core Principle

角色类文档优先讲清角色边界，而不是只围绕单一动作展开。

### Recommended Structure

1. 角色说明
2. 支持登录的端
3. 登录或进入方式
4. 默认能力
5. 额外授权能力
6. 角色限制

### Role Writing Rules

- 先讲“这个角色是谁、主要负责什么”。
- 角色的主要任务可以突出，但不能把角色能力写窄成单一动作。
- 多端时需要覆盖所有可登录端，但不必机械地把每个端从头到尾完整重写一遍。
- 多端内容优先采用“通用能力 + 端差异”的写法。
- 若某个端有独有能力或明显交互差异，应单独写出。
- 额外授权能力应明确标出由谁授予、授予后会增加什么能力。
- 角色限制要写明确，不要只写抽象权限描述。应让用户知道：
  - 默认能做什么。
  - 默认不能做什么。
  - 哪些能力需要他人授权。
  - 不同端支持范围可能不同。
- 不需要为了对比而强行制作表格；自然分段说明即可。

### Common Review Checks For Role Docs

- 是否讲清角色是谁、主要职责是什么。
- 是否把角色能力写窄成单一动作。
- 是否区分默认能力、授权能力和角色限制。
- 是否说明了多端差异。
- 是否明确指出哪些操作需要他人授权或由其他角色处理。

## FAQ Documents

适用场景：用户已经遇到具体问题，希望快速找到原因与解决方案。

### Recommended Structure

1. 问题是什么
2. 可能原因是什么
3. 解决方案是什么
4. 如仍未解决，下一步看哪里

### FAQ Writing Rules

- 标题尽量贴近用户真实会搜索的问题。
- 先回答问题，再解释背景。
- 原因说明尽量按最常见、最可验证的情况排序。
- 解决方案要可执行，尽量写清入口、检查项和处理动作。
- 若问题本质上需要完整教程，FAQ 只保留问题导向说明，并挂到对应功能文档。
- FAQ 的重点是快速解决问题，不是复述产品介绍。

### Common Review Checks For FAQ Docs

- 标题是否站在客户视角。
- 是否先回答问题，而不是先讲功能背景。
- 解决方案是否可执行。
- 是否需要分流到更完整的帮助文档。
